12. 성능 시험

12.1. DB 변경 성능 영향도 분석

개요

데이터베이스 업그레이드, U2L(Unix To Linux) 등의 작업은 SQL문의 실행계획이 변경되어 SQL의 성능이 저하될 수 있다. 데이터베이스 변경으로 인한 SQL의 성능 변화를 간단하고 손쉽게 시뮬레이션하여, 성능 개선이 필요한 SQL을 튜닝대상으로 선정하고 개선할 수 있는 기능을 제공한다.

화면 구성

화면은 다음과 같은 탭으로 구성되어 있다.

Tab 명 Tab 설명
성능 영향도 분석
  • ASIS DB(원천DB)에서 운영 중인 SQL을 수집하여 TOBE DB(목표DB)에 자동 실행한 후, ASIS 성능과 TOBE 성능 정보를 비교하여 저장한다.
성능 영향도 분석 결과
  • ASIS 성능과 TOBE 성능 비교 결과를 다양한 조건으로 검색할 수 있으며, 성능 저하로 인해 튜닝이 필요한 SQL에 대해 Profile을 고정하거나 튜닝대상으로 선정할 수 있다.
튜닝실적
  • 성능 저하로 인해 튜닝 대상으로 선정된 SQL에 대한 튜닝 진행 현황 및 개선 실적 정보를 제공한다.
튜닝 SQL 일괄 검증
  • 튜닝이 완료된 SQL에 대한 성능을 일괄 검증한다. ASIS, TOBE, TUNING, 일괄검증 각각의 성능 정보를 비교하여 저장한다.
운영 SQL 성능 추적
  • ASIS DB(원천DB)에서 수집된 SQL의 성능과 새롭게 오픈한 운영 DB의 성능을 비교 추적하여, ASIS 대비 성능이 저하된 SQL추적 및 튜닝대상으로 선정하여 개선할 수 있다.
SQL 점검팩
  • SQL점검팩을 등록/변경/삭제할 수 있는 기능을 제공한다.
[성능 영향도 분석] 탭

성능 영향도 분석을 수행 할 SQL점검팩을 [SQL점검팩] 탭에서 등록 후, 수집 대상 ASIS DB의 수집기간 범위 및 필터링 조건(전체 SQL 또는 TOP SQL, OWNER/MODULE)을 입력한다. TOBE DB를 선택 후 [실행] 버튼을 클릭하면 성능 영향도 분석 작업이 자동 수행된다.
성능 영향도 분석 수행 시 ASIS DB의 SQL문, 바인드 값, 실행성능, 플랜정보를 수집한다. 수집된 SQL문장과 바인드 값으로 TOBE DB에 실행 후 TOBE DB의 실행성능, 플랜정보를 ASIS DB와 비교하여 저장한다.

성능 영향도 분석 실행

  1. 프로젝트 > SQL점검팩을 선택한다. 신규 SQL점검팩 일 경우 상세 조건을 입력한 후 [실행] 버튼을 클릭한다.
    컬럼 컬럼 설명
    ASIS DB(원천DB)
    • 성능 영향도 분석을 수행을 위한 SQL을 수집할 DB를 선택한다
    TOBE DB(목표DB)
    • 성능 영향도 분석을 수행할 TOBE DB를 선택한다.
    수집기간
    • ASIS DB에서 SQL을 수집할 기간을 선택한다.
    AWR
    • 체크 시 AWR을 대상으로 성능 영향도 분석을 수행한다.
    전체 SQL
    • 체크 시 V$를 대상으로 성능 영향도 분석을 수행한다.
    전체
    • 체크 시 수집된 SQL 전체에 대한 성능 영향도 분석을 수행한다.
    • 체크 해제 시 TOP N을 입력할 수 있다.
    TOP N
    • 수집된 SQL 중 TOP N에 입력된 상위 N개의 SQL에 대해서만 성능 영향도 분석을 수행한다.(TOP N의 선정 기준은 BUFFER GETS 기준)
    • 전체 체크 해제 시 TOP N을 입력할 수 있다.
    Literal 제외
    • 체크 시 Literal SQL을 제외한다.
    • 체크 해제 시 Literal SQL을 포함한다.
    TABLE_OWNER in
    • ASIS DB에서 특정 OWNER에 대한 수집이 필요한 경우 입력한다.(옵션)
    • N개의 OWNER를 입력할 경우 콤마(,)로 분리하며 대소문자는 구분하지 않으며 최대 100개까지 입력 가능하다.
      예) ORACLE, OpenPOP, orange
    • 입력된 OWNER와 동일한 SQL에 대해서만 수집한다. 입력된 OWNER는 내부적으로 아래와 같은 조건으로 필터링에 포함된다.
      WHERE UPEER(OWNER) IN (‘ORACLE’, ‘OPENPOP’, ORANGE’)
    MODULE like
    • ASIS DB에서 특정 MODULE에 대한 수집이 필요한 경우 입력한다.(옵션)
    • N개의 MODULE을 입력할 경우 콤마(,)로 분리하며 대소문자는 구분하지 않으며 최대 100개까지 입력 가능하다.
      예) JDBC, dbsnmp
    • 입력된 MODULE이 포함된 SQL에 대해서만 수집한다. 입력된 MODULE은 내부적으로 아래와 같은 조건으로 필터링에 포함된다.
      WHERE (UPEER(MODULE) LIKE ‘%JDBC%’ OR
      UPEER(MODULE) LIKE ‘%DNSMP%’)
    Filter SQL

    Filter SQL

    • [Filter SQL] 버튼을 클릭한다.
    • Filter SQL 팝업 창이 뜨고, Module, Action, Parsing Schema Name, SQL_TEXT, SQL ID 항목에 대한 SQL을 추가한 후 [저장]버튼을 클릭한다.
    • 저장된 내용이 [Filter SQL] 버튼 옆에 보여진다.
    • SQL_TEXT 선택 시 LIKE , NOT LIKE 만 선택가능하다.
    실행방법
    • Plan비교 후 변경SQL실행, SQL실행, Plan변경 비교 중 선택한다.
      • Plan비교 후 변경SQL실행: 수집된 SQL에 대해 먼저 PLAN 생성 후, PLAN 변경된 SQL을 실행하여 성능 비교
      • SQL실행: 수집된 SQL을 실행하여 성능 비교
      • Plan변경 비교: 수집된 SQL의 PLAN 생성 후 PLAN 변경 여부 비교
    • Plan변경 비교를 선택한 경우 병렬실행만 변경이 가능하다
    병렬실행
    • 동시에 실행 가능한 프로세스 개수를 선택한다.
    • 버튼을 눌러서 YES로 선택 후 프로세스 개수를 입력한다.
    • 1부터 최대 8까지 입력이 가능하다.
    DML실행
    • SQL실행 시 DML문을 실행할 것인지 선택한다.
      • YES: DML문을 실행
      • NO: 예측 플랜 생성
    Multiple실행
    • SQL 실행 시 동일 SQL을 몇 번 실행할 것인지 선택한다.
    • 1부터 최대 10까지 입력이 가능하다.
    Multiple Bind 실행
    • SQL 실행 시 동시에 Bind종류를 몇 개까지 실행할 것인지 선택한다.
    • 1부터 최대 10까지 입력이 가능하다
    SQL Time Limit
    • SQL 타임 제한 기능을 선택한다. 선택된 시간을 초과한 SQL에 대해서는 실행을 중단하고 다음 SQL을 실행한다.
    • ‘직접입력’을 선택할 경우 사용자가 직접 제한 시간을 입력할 수 있다.
    최대 Fetch 건수
    • 최대 Fetch 건수를 입력한다. 입력된 Fetch 건수를 초과한 SQL에 대해서는 실행을 중단하고 다음 SQL을 실행한다.
  2. 진행상황을 수행결과에서 조회한다.
  3. [실행] 버튼 클릭 시 SQL수집 건수가 0건일 경우 아래와 같은 메시지 박스가 팝업되며, 조건값을 수정하여 성능 영향도 분석을 수행할 수 있다.
    [실행] 버튼 클릭 시 해당 SQL점검팩에 대한 성능 영향도 분석 작업이 수행 중일 경우, 아래와 같은 메시지 박스가 팝업된다.
  4. 아이콘 클릭 시 입력된 값을 상세히 볼 수 있다.

NOTE

성능 영향도 분석 작업은 백그라운드로 실행된다. 따라서 실행 후 다른 작업을 하거나, Open-POP WEB이 종료되더라도 재접속 후 해당 SQL점검팩의 진행상황을 조회할 수 있다.

성능 영향도 분석 재수행

TOBE DB의 환경변화에 따라 이전에 수행된 SQL점검팩을 재수행 할 수 있다. 재수행 시 이전에 수행된 성능 영향도 분석 정보는 물리적으로 삭제되므로 주의가 필요하다.
실행영역에 병렬실행, DML실행, Multiple실행, Multiple Bind실행은 변경이 가능하다. (단, 실행방법이 ‘Plan 변경 비교’일 경우에는 병렬실행만 변경이 가능하다.)

NOTE

이미 수행된 SQL 점검팩의 상세 조건은 ASIS DB의 성능 영향도 분석 대상 범위(모수)를 명확하게 하기 위해 수정할 수 없다. 수정이 필요할 경우 새로운 SQL점검팩을 신규 등록 후 성능 영향도 분석을 수행한다. 불필요한 SQL점검팩의 경우 [SQL점검팩] 탭에서 삭제한다.
튜닝대상이 선정되었을 경우 성능 영향도 분석을 재수행 할 수 없다. 재수행 시 모든 데이터가 삭제되므로 실수로 인한 데이터 삭제를 방지하기 위한 목적이다.

  1. 재수행 할 프로젝트 > SQL점검팩을 선택하면 이전에 수행된 SQL점검팩의 상세 조건이 조회된다.
  2. [실행] 버튼 클릭 시 아래와 같은 메시지 박스가 팝업되고 [YES] 버튼 클릭 시 성능 영향도 분석 작업이 재수행 된다.
    튜닝대상으로 선정된 SQL이 존재하는 경우 아래와 같은 메시지 박스가 팝업되고, SQL점검팩을 삭제할 수 없다.

강제완료처리

진행 중인 성능 영향도 분석 작업을 중단해야 할 사유 발생 시 강제완료처리를 수행한다.

  1. 수행 중인 SQL점금팩에서 [강제완료처리] 버튼을 클릭한다
  2. 아래와 같은 메시지 박스가 팝업되고 [YES] 버튼 클릭 시 작업이 강제완료 된다.
    해당 SQL점검팩에 수행 중인 작업이 없을 경우 아래와 같은 메시지 박스가 팝업된다.
  3. 강제완료처리 시 수행결과에 아래와 같이 조회된다.

권한

Open-POP의 USER로 성능 영향도 분석을 수행하기 때문에 TOBE DB에 권한이 부여되지 않아 정상적으로 수행할 수 없는 경우가 발생한다. 이때 필요한 권한 스크립트를 자동으로 생성할 수 있는 기능을 제공한다. 생성된 스크립트를 TOBE DB에 실행한 후 재수행한다.

  1. [권한] 버튼을 클릭한면 권한스크립트 화면이 실행된다.
  2. Object Owner 선택 > Object 선택 > [권한스크립트 생성] 버튼 클릭 > [권한스크립트 복사] 버튼을 클릭한다.
  3. [권한스크립트 생성] 버튼을 클릭하면 선택된 Object에 대한 권한 스크립트가 생성되고, [권한스크립트 복사] 버튼을 클릭하면 클립보드에 복사된다. 복사된 스크립트를 TOBE DB에 실행한 후 재수행한다.

  • ASIS와 비교하여 TOBE에서 BUFFER GET이 입력된 값 이상 성능이 저하된 SQL을 검색한다. (소수점 입력 가능)
ELAPSED TIME 성능 저하
  • ASIS와 비교하여 TOBE에서 ELAPSED TIME 이 입력된 값 이상 성능이 저하된 SQL을 검색한다. (소수점 입력 가능)
SQL Profile 적용 / 미적용
  • Profile이 적용된 SQL을 검색한다.
  • 적용과 미적용 다 체크 시 전체가 조회된다.
SQL 검토 / 미검토
  • SQL 검토가 등록된 SQL을 검색한다.
  • 검토 체크 시 키워드 입력이 가능하며, 키워드가 포함된 SQL을 전체 조회한다.
  • 키워드가 없을 경우 검토가 등록된 SQL을 전체 조회한다.
  • 검토과 미검토 다 체크 시 전체가 조회된다.

튜닝대상 선정 전체 선택

검색된 전체 SQL(모든 페이지)에 대해 튜닝대상으로 선정한다. 선정된 튜닝대상에 대한 튜닝 작업은 ‘성능 최적화’ 메뉴에서 진행한다.

  2. [튜닝대상선정] 버튼을 클릭한다. 튜닝 요청 화면이 실행되고 튜닝담당자, 이전 튜닝대상 선정 SQL 제외 항목을 선택하고 [튜닝요청 및 담당자 지정] 버튼을 클릭한다.
    컬럼 컬럼 설명
    튜닝담당자 자동할당
    • ‘Yes’일 경우 튜닝담당자를 자동할당 한다.
    튜닝담당자
    • 튜닝담당자 자동할당이 ‘No’일 경우 튜닝담당자를 직접 선택한다.
    이전 튜닝대상 선정 SQL 제외
    • ‘Yes’일 경우 해당 SQL점검팩에서 튜닝대상 선정이 된 적이 있는 SQL은 튜닝대상에서 제외한다.
    프로젝트
    • 프로젝트
    SQL점검팩
    • SQL점검팩
  3. 튜닝대상 선정 작업이 진행되고 완료되면 아래와 같은 메시지 박스가 팝업된다. 메시지 내용에 선정된 튜닝대상 건수를 확인할 수 있다.
  4. [Ok] 버튼을 클릭하면 그리드가 재조회 되며 튜닝상태, 튜닝번호 컬럼에서 확인이 가능하다.

  1. 튜닝대상 SQL에 대한 체크 박스를 클릭한다. 멀티 선택이 가능하다.
  2. 이후의 처리 과정은 ‘> 튜닝대상 선정 전체 선택’ 과 처리 과정이 동일하므로 해당 매뉴얼을 참조한다.
SQL Profile 적용/삭제/이관

SQL Profile 적용 전체

검색된 전체 SQL(모든 페이지)에 대해 Profile을 적용한다.

  1. ‘전체’ 체크 박스를 클릭하면 하위의 모든 체크 박스가 체크된다.
  2. [SQL Profile 적용] 버튼을 클릭한다. SQL Profile 적용 화면이 실행되고 [SQL Profile 적용] 버튼을 클릭한다. 화면의 “적용대상 Profile”에서 Profile 적용되는 SQL수를 확인할 수 있다.
  3. SQL Profile 적용 작업이 진행되고 완료되면 아래와 같은 메시지 박스가 팝업된다. 메시지에서 대상 건수 및 성공 건수를 확인할 수 있다.

SQL Profile 적용 개별

검색된 SQL 중 개별 선택된 건에 대해 TOBE DB에 Profile을 적용한다. 개별 선택은 현재 페이지에서만 가능하다. [다음] 버튼을 클릭하여 페이지가 변경되면 선택된 체크박스가 해제된다.

  1. Profile 적용 대상 SQL에 대한 체크 박스를 클릭한다. 멀티 선택이 가능하다.
  2. 이후의 처리 과정은 ‘> SQL Profile 적용 전체’ 와 처리 과정이 동일하므로 해당 매뉴얼을 참조한다.

SQL Profile 삭제

적용된 Profile을 삭제한다.

  1. SQL Profile 적용된 SQL을 검색한다. 검색 조건에서 ‘SQL Profile 적용’을 체크하고 [검색] 버튼을 클릭한다.
  2. Profile 적용 대상 SQL에 대한 체크 박스를 클릭한다. 멀티 선택이 가능하다. 또한 전체를 체크할 경우 전체 선택이 가능하다.
  3. [SQL Profile 삭제] 버튼을 클릭한다. SQL Profile 삭제 화면이 실행되고 [SQL Profile 삭제] 버튼을 클릭한다. 화면의 “삭제대상 Profile”에서 Profile이 삭제되는 SQL수를 확인할 수 있다.
  4. SQL Profile 삭제 작업이 진행되고 완료되면 아래와 같은 메시지 박스가 팝업된다. 메시지에서 대상 건수 및 성공 건수를 확인할 수 있다.

SQL Profile 이관

TOBE DB에 적용된 Profile을 이관대상DB로 이관한다.

  1. SQL Profile 적용된 SQL을 검색한다. 검색 조건에서 ‘SQL Profile 적용’을 체크하고 [검색] 버튼을 클릭한다.
  2. Profile 적용 대상 SQL에 대해 전체를 체크한다. 개별적으로 이관을 할 경우 각 행의 체크 박스를 클릭한다. 멀티 선택이 가능하다.
  3. [SQL Profile 이관] 버튼을 클릭한다. SQL Profile 이관 화면이 실행되고, “이관대상DB” 선택 > [SQL Profile 이관] 버튼을 클릭한다. 화면의 “이관대상 Profile”에서 Profile이 이관되는 SQL수를 확인할 수 있다.
  4. SQL Profile 이관 작업이 진행되고 완료되면 아래와 같은 메시지 박스가 팝업된다.
성능 영향도 분석 재실행

NOTE

성능 영향도 분석 결과 에서 조회된 SQL에 대해서 성능 영향도 분석 작업을 재 실행 한다. 재 실행이 완료 될 때까지 다른 작업은 할수 없고, 재 실행 완료 후 List가 자동으로 재 조회 된다. 전체 선택은 현재 페이지에서 조회된 전체 건 수에 한해서만 성능 영향도 분석 재실행이 가능하다.

성능 영향도 분석 재실행 개별선택

  1. SQL Profile 적용된 SQL을 검색한다. 검색 조건에서 ‘SQL Profile 적용’을 체크하고 [검색] 버튼을 클릭한다.
  2. Profile 적용 대상 SQL에 대해 전체를 체크한다. 개별적으로 이관을 할 경우 각 행의 체크 박스를 클릭한다. 멀티 선택이 가능하다.

성능 영향도 분석 재실행 전체선택

  1. SQL Profile 적용된 SQL을 검색한다. 검색 조건에서 ‘SQL Profile 적용’을 체크하고 [검색] 버튼을 클릭한다.
  2. Profile 적용 대상 SQL에 대해 전체를 체크한다. 개별적으로 이관을 할 경우 각 행의 체크 박스를 클릭한다. 멀티 선택이 가능하다.
검색결과 건수

[검색] 버튼 클릭 시 해당 조건으로 검색되는 전체 결과 건수를 표시한다.

그리드Area 최대화

성능 영향도 분석이 완료된 데이터를 조회하고, List를 최대화하여 여러 데이터를 한번에 확인할 수 있게 한다.
최대화 버튼 클릭 시 검색조건,수행결과 & 차트Area가 사라지고 그리드Area가 최대화된다.

  1. 우측 하단에 최대화 Icon을 클릭한다.
  2. 검색조건Area, 수행결과 & 차트 Area가 사라지고, 아래와 같이 그리드 Area가 최대화된다.
  3. 최대화Icon을 다시 클릭하면 아래와 같이 처음화면으로 되돌아온다.
[튜닝 실적] 탭

ASIS 성능과 TOBE 성능 및 TUNING 후의 성능 정보가 SQL별로 제공되어 성능의 개선 정보를 한눈에 볼 수 있다.

[SQL 점검팩] 탭

SQL점검팩을 등록/수정/삭제할 수 있는 기능을 제공한다. SQL점검팩 삭제 시 SQL점검팩 및 수행된 성능 영향도 분석 정보도 모두 삭제되므로 주의가 필요하다.

NOTE

성능 영향도 분석 작업이 완료되고 튜닝대상을 선정한 SQL점검팩에 대해서는 삭제할 수 없다.
SQL점검팩을 삭제할 경우 튜닝 실적 등 성능 영향도 분석 작업 완료 후 수행된 모든 정보를 삭제하므로, 사용자 실수로 인한 삭제를 방지하기 위한 목적이다.

SQL점검팩 등록

  1. [등록/수정] 버튼을 클릭한다.
  2. SQL점검팩 화면이 실행되며, 관련 정보를 입력하고 [저장] 버튼을 클릭한다.
    다른 방법으로, [초기화] 버튼 클릭 시 입력 된 정보가 초기화되며 신규 등록이 가능하다.

SQL점검팩 수정

  1. 수정할 SQL점검팩을 그리드에서 클릭하고, [등록/수정] 버튼을 클릭한다.
  2. SQL점검팩 화면이 실행되며 해당 SQL점검팩의 정보가 조회된다. 수정할 정보를 입력하고 [저장] 버튼을 클릭한다.
    다른 방법으로, SQL점검팩 화면에서 수정할 SQL점검팩을 선택한 후 수정할 수 있다.
    [초기화] 버튼 클릭 시 입력 된 정보가 초기화되며 신규 등록이 가능하다.

SQL점검팩 삭제

  1. [검색] 후 그리드에서 삭제할 SQL점검팩을 체크한 후 [삭제] 버튼을 클릭한다.
  2. 아래와 같은 메시지 박스가 팝업되고 [YES] 버튼 클릭 시 SQL점검팩이 삭제되었다는 메시지 박스가 팝업된다. SQL점검팩 삭제시 성능 영향도 분석 정보가 모두 삭제되므로 주의가 필요하다.
    튜닝대상으로 선정된 SQL이 존재하는 경우 아래와 같은 메시지 박스가 팝업되고, SQL점검팩을 삭제할 수 없다.

NOTE

튜닝대상이 선정되었을 경우 SQL점검팩을 삭제할 수 없다. 삭제 시 모든 데이터가 삭제되므로 실수로 인한 데이터 삭제를 방지하기 위한 목적이다.
