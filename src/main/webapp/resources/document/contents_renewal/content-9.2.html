9. 성능 검증

9.2. 배포전 자동 성능 검증

개요

배포 시스템과 연계하여 프로그램 배포 시 SQL에 대한 성능을 자동 검증한다. 성능 검증은 테스트 서버에서 테스트 한 SQL의 성능 정보를 Open-POP이 수집하여 프로그램 배포 요청 시, 수집된 SQL과 배포된 SQL을 비교하여 성능을 검증하고 그 결과를 배포 시스템으로 통보한다. 배포되는 모든 SQL에 대한 성능을 검증하는 것이 아니라 변경된 SQL에 대해서만 성능 검증을 수행한다.

화면 구성

화면은 다음과 같은 탭으로 구성되어 있다.

성능 검증 관리

예외 요청
  • 부적합/미수행 프로그램에 대하여 필요할 경우 성능 예외를 요청하고 처리한다.
예외 삭제
  • 예외 처리된 건에 대한 삭제가 필요할 경우 예외 삭제를 요청하고 처리한다.
예외 처리 이력
  • SQL에 대한 예외 처리 이력을 제공한다. 특정 SQL식별자(DBIO)의 예외 처리와 관련한 추적이 가능하다.
[성능 검증 관리] 탭

배포 단위의 성능 검증 대상 목록 및 성능 검증 결과를 제공한다. 미수행 된 건이 포함된 경우 재배포 없이 성능 검증을 재수행할 수 있으며, 시스템 상의 원인으로 배포 시스템에 자동 통보되지 않았을 경우 검증 결과를 재통보할 수 있다.

NOTE

다음과 같은 경우 배포 시스템에 자동 통보된다.

  1. 성능 검증 결과 적합일 경우
  2. 배포된 SQL이 변경이 없을 경우

버튼 기능

버튼 설명
검증결과 통보
  • 검증완료 된 건에 대해 배포시스템으로 성능검증 결과를 통보한다.
  • 정상적으로 성능검증을 완료했으나, 시스템의 오류 등의 사유로 배포시스템에 전송되지 않았을 경우에만 사용한다.
강제 검증완료
  • 시스템의 오류 등 강제완료 처리 사유 발생 시 사용한다.
  • 강제완료 처리 시 배포시스템에 적합으로 통보된다.
성능검증 재수행
  • 미수행 건 존재 시 배포 시스템에서 응용 프로그램을 재테스트 후 성능검증을 재수행한다. 미수행 건이 존재한다는 것은 해당 SQL이 테스트 시 수행되지 않았다는 것을 의미한다.
  • 재수행 시 해당 배포ID에 대한 이전 성능검증 결과가 모두 삭제되고, 재수행 시점까지의 테스트 결과에 대해 성능검증을 재수행 한다.
성능 검증 결과

배포 단위의 SQL별 성능 검증 결과를 제공한다. 검증결과 ‘부적합’ 및 ‘미수행’ 건일 경우 [예외요청] 버튼이 활성화되어 예외요청을 처리할 수 있으며, ‘검증중’인 건에 대해 성능 검증을 완료 처리하고 배포시스템에 통보할 수 있다.

NOTE

성능검증 평가 방법 설정-‘평균’ 또는 ‘최대’ 배포되는 SQL이 Dynamic SQL일 경우 SQL식별자(DBIO) 당 여러 개의 SQL ID가 존재한다. SQL ID 별로 성능 결과 값이 상이하므로 Open-POP은 ‘평균’, ‘최대’ 2가지 평가 방법을 제공한다. 고객사는 2가지 방법 중 하나를 선택할 수 있다.

  • 평균: SQL식별자(DBIO)를 기준으로 수행된 SQL들의 평균값을 적합 기준값과 비교하여 성능 검증 수행
    • 검증지표 중 SQL수행시간(초), 블럭수, 처리건수, 메모리사용량(MB)은 각 지표의 평균값을 구함
    • 단, 검증지표 대용량 테이블 Full Scan 여부, 전체 파티션 엑세스 여부가 1건이라도 있으면 부적합 처리
  • 최대: SQL식별자(DBIO)를 기준으로 수행된 SQL들의 최대값을 적합 기준값과 비교하여 성능 검증 수행
    • 검증지표 중 SQL수행시간(초), 블럭수, 처리건수, 메모리사용량(MB)은 각 지표의 최대값을 구함
    • 단, 검증지표 대용량 테이블 Full Scan 여부, 전체 파티션 엑세스 여부가 1건이라도 있으면 부적합 처리


예시) ‘평균’ 성능검증 평가방법 화면 – ‘적합’ 값과 ‘평균’ 값을 비교하여 성능 검증
예시) ‘최대’ 성능검증 평가방법 화면 – ‘적합’ 값과 ‘최대’ 값을 비교하여 성능 검증
