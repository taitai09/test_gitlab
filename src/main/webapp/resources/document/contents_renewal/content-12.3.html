12. 성능 시험

12.3. 테이블 변경 성능 영향도 분석

개요

인덱스 추가, 파티셔닝 등의 구조변경에 대하여 해당 테이블을 참조하는 SQL의 영향도를 분석한다. 운영 DB 반영 전에 검증 DB에서 성능의 변화를 진단하여, 테이블 변경으로 인한 운영DB 장애를 최소화 할 수 있는 기능을 제공한다.

화면 구성

화면은 다음과 같은 탭으로 구성되어 있다.

Tab 명 Tab 설명
성능 영향도 분석
  • 소스 DB에서 운영 중인 SQL을 수집하여 목표DB에 자동 실행한 후, ASIS 성능과 TOBE 성능 정보를 비교하여 저장한다.
성능 영향도 분석 결과
  • 소스DB 성능과 목표DB 성능 비교 결과를 다양한 조건으로 검색할 수 있으며, 성능 저하로 인해 튜닝이 필요한 SQL에 대해 튜닝대상으로 선정할 수 있다.
튜닝실적
  • 성능 저하로 인해 튜닝 대상으로 선정된 SQL에 대한 튜닝 진행 현황 및 개선 실적 정보를 제공한다.
SQL 점검팩
  • SQL점검팩을 등록/변경/삭제할 수 있는 기능을 제공한다.
[성능 영향도 분석] 탭

성능 영향도 분석을 수행 할 SQL점검팩을 [SQL점검팩] 탭에서 등록 후, 수집 대상 소스 DB, 목표DB 선택과 수집기간 범위 및 필터링 조건(AWR 또는 전체 SQL, TOP SQL, OWNER/ TABLE/MODULE)을 입력한다. [실행] 버튼을 클릭하면 성능 영향도 분석이 자동 수행된다.
성능 영향도 분석 수행 시 소스 DB의 SQL문, 바인드 값, 실행성능, 플랜정보를 수집한다. 수집된 SQL문장과 바인드 값으로 목표 DB에 실행 후 목표 DB의 실행성능, 플랜정보를 소스 DB와 비교하여 저장한다.

NOTE

SELECT문은 가장 최근의 바인드 값으로 SQL을 실행하여 수행 시간과 블록수 기준으로 성능을 비교한다.
DML 문은 바인드 값이 수집되지 않기 때문에 예측 실행플랜을 생성하여 플랜 변경 여부와 플랜상의 비효율적인 동작, 즉 테이블 FULL스캔 여부, 전체 파티션 엑세스 여부가 존재하는지를 점검한다.

성능영향도 분석 실행 Area

성능 영향도 분석 실행

  1. 신규 일 경우 SQL점검팩을 [SQL점검팩] 탭에서 등록한다. 등록하는 방법은 [SQL점검팩] 탭의 설명을 참조한다.
  2. 점검팩 등록일자 선택 > SQL점검팩을 선택한다. 신규 SQL점검팩 일 경우 상세 조건을 입력한 후 [실행] 버튼을 클릭한다.
    컬럼 컬럼 설명
    AWR
    • 체크 시 AWR을 대상으로 성능 영향도 분석을 수행한다.
    전체 SQL
    • 체크 시 V$를 대상으로 성능 영향도 분석을 수행한다.
    전체
    • 체크 시 수집된 SQL 전체에 대한 성능 영향도 분석을 수행한다.
    • 체크 해제 시 TOP N을 입력할 수 있다.
    TOP N
    • 수집된 SQL 중 TOP N에 입력된 상위 N개의 SQL에 대해서만 성능 영향도 분석을 수행한다.(TOP N의 선정 기준은 BUFFER GETS 기준)
    • 전체 체크 해제 시 TOP N을 입력할 수 있다.
    소스DB
    • 성능 영향도 분석을 수행을 위한 SQL을 수집할 DB를 선택한다.
    목표DB
    • 성능 영향도 분석을 수행할 목표DB를 선택한다.
    수집기간
    • 소스 DB에서 SQL을 수집할 기간을 선택한다.
    Literal 제외
    • 체크 시 Literal SQL을 제외한다.
    • 체크 해제 시 Literal SQL을 포함한다.
    SQL Time Limit
    • SQL 타임 제한 기능을 선택한다. 선택된 시간을 초과한 SQL에 대해서는 실행을 중단하고 다음 SQL을 실행한다.
    • ‘직접입력’을 선택할 경우 사용자가 직접 제한 시간을 입력할 수 있다.
    최대 Fetch 건수
    • 최대 Fetch 건수를 입력한다. 입력된 Fetch 건수를 초과한 SQL에 대해서는 실행을 중단하고 다음 SQL을 실행한다.
    TABLE_OWNER
    • 소스DB의 TABLE OWNER를 선택한다.
    TABLE_NAME in
    • 소스 DB에서 성능 영향도 분석을 수행할 TABLE을 입력한다.
    • N개의 TABLE을 입력할 경우 콤마(,)로 분리하며 대소문자는 구분하지 않으며 최대 100개까지 입력 가능하다.
      예) USERS, menu, role
    • 입력된 TABLE과 동일한 TABLE을 사용하는 SQL에 대해서만 수집한다. 입력된 TABLE은 내부적으로 아래와 같은 조건으로 필터링에 포함된다.
      WHERE UPEER(TABLE_NAME) IN (‘USERS’, ‘MENU’, ‘ROLE’)
    MODULE like
    • 소스 DB에서 특정 MODULE에 대한 수집이 필요한 경우 입력한다.(옵션)
    • N개의 MODULE을 입력할 경우 콤마(,)로 분리하며 대소문자는 구분하지 않으며 최대 100개까지 입력 가능하다.
      예) JDBC, dbsnmp
    • 입력된 MODULE이 포함된 SQL에 대해서만 수집한다. 입력된 MODULE은 내부적으로 아래와 같은 조건으로 필터링에 포함된다.
      WHERE (UPEER(MODULE) LIKE ‘%JDBC%’ OR
      UPEER(MODULE) LIKE ‘%DNSMP%’)
  3. 진행상황을 수행결과에서 조회한다.
    [실행] 버튼 클릭 시 SQL수집 건수가 0건일 경우 아래와 같은 메시지 박스가 팝업되며, 조건값을 수정하여 성능 영향도 분석을 수행할 수 있다.
    [실행] 버튼 클릭 시 해당 SQL점검팩에 대한 성능 영향도 분석 작업이 수행 중일 경우, 아래와 같은 메시지 박스가 팝업된다.
    아이콘 클릭 시 입력된 값을 상세히 볼 수 있다.

NOTE

성능 영향도 분석 작업은 백그라운드로 실행된다. 따라서 실행 후 다른 작업을 하거나, Open-POP WEB이 종료되더라도 재접속 후 해당 SQL점검팩의 진행상황을 조회할 수 있다.

성능 영향도 분석 재수행

목표 DB의 환경변화에 따라 이전에 수행된 SQL점검팩을 재수행 할 수 있다. 재수행 시 이전에 수행된 성능 영향도 분석 정보는 물리적으로 삭제되므로 주의가 필요하다.

NOTE

이미 수행된 SQL의 점검팩의 상세 조건은 소스 DB의 성능 영향도 분석 대상 범위를 명확하게 하기 위해 수정할 수 없다. 수정이 필요할 경우 새로운 SQL점검팩을 신규 등록 후 성능 영향도 분석을 수행한다. 불필요한 SQL점검팩의 경우 [SQL점검팩] 탭에서 삭제한다.
튜닝대상이 선정되었을 경우 성능 영향도 분석을 재수행 할 수 없다. 재수행 시 모든 데이터가 삭제되므로 실수로 인한 데이터 삭제를 방지하기 위한 목적이다.

  1. 재수행 할 SQL점검팩을 선택하면 이전에 수행된 SQL점검팩의 상세 조건이 조회된다.
  2. [실행] 버튼 클릭 시 아래와 같은 메시지 박스가 팝업되고 [YES] 버튼 클릭 시 성능 영향도 분석 작업이 재수행 된다.
    튜닝대상으로 선정된 SQL이 존재하는 경우 아래와 같은 메시지 박스가 팝업되고, SQL점검팩을 삭제할 수 없다.

강제완료처리

진행 중인 성능 영향도 분석 작업을 중단해야 할 사유 발생 시 강제완료처리를 수행한다.

  1. 수행 중인 SQL점금팩에서 [강제완료처리] 버튼을 클릭한다
  2. 아래와 같은 메시지 박스가 팝업되고 [YES] 버튼 클릭 시 성능 영향도 분석 작업이 강제완료 된다.
    해당 SQL점검팩에 수행 중인 성능 영향도 분석 작업이 없을 경우 아래와 같은 메시지 박스가 팝업된다.
  3. 강제완료처리 시 수행결과에 아래와 같이 조회된다.

권한

Open-POP의 USER로 성능 영향도 분석을 수행하기 때문에 목표DB에 권한이 부여되지 않아 성능 영향도 분석을 정상적으로 수행할 수 없는 경우가 발생한다. 이때 필요한 권한 스크립트를 자동으로 생성할 수 있는 기능을 제공한다. 생성된 스크립트를 목표DB에 실행한 후 성능 영향도 분석을 재수행한다.

  1. [권한] 버튼을 클릭한면 권한스크립트 화면이 실행된다.
  2. Object Owner 선택 > Object 선택 > [권한스크립트 생성] 버튼 클릭 > [권한스크립트 복사] 버튼을 클릭한다.
  3. [권한스크립트 생성] 버튼을 클릭하면 선택된 Object에 대한 권한 스크립트가 생성되고, [권한스크립트 복사] 버튼을 클릭하면 클립보드에 복사된다. 복사된 스크립트를 목표 DB에 실행한 후 성능 영향도 분석을 재수행한다.

[성능 영향도 분석 결과] 탭

소스DB 성능과 목표DB 성능 영향도 분석 결과를 다양한 조건으로 검색할 수 있으며, 성능 저하로 인해 튜닝이 필요한 SQL에 대해 튜닝대상으로 선정할 수 있다.
ELAPSED TIME, BUFFER GETS, FULL SCAN여부, PARTITON ALL ACCESS여부 등 성능 영향도 분석 결과에 대해 소스 DB와 목표DB를 비교하여 제공하며, 버퍼 임팩트(%), 수행시간 임팩트(%), PLAN변경여부 및 성능점검결과가 적합인지 부적합인지에 대한 정보를 제공한다.

튜닝대상 선정

튜닝대상 선정 전체 선택

검색된 전체 SQL(모든 페이지)에 대해 튜닝대상으로 선정한다. 선정된 튜닝대상에 대한 튜닝 작업은 ‘성능 최적화’ 메뉴에서 진행한다.

  1. ‘전체’ 체크 박스를 클릭하면 하위의 모든 체크 박스가 체크된다.

    NOTE

    ‘전체 체크’ 박스를 선택할 경우 현재 그리드에 조회된 SQL뿐만 아니라, 검색된 조건의 모든 SQL(다음 페이지 포함)에 대해 튜닝대상으로 선정되므로 주의가 필요하다.

  2. [튜닝대상선정] 버튼을 클릭한다. 튜닝 요청 화면이 실행되고 튜닝담당자, 이전 튜닝대상 선정 SQL 제외 항목을 선택하고 [튜닝요청 및 담당자 지정] 버튼을 클릭한다.
    컬럼 컬럼 설명
    튜닝담당자 자동할당
    • ‘Yes’일 경우 튜닝담당자를 자동할당 한다.
    튜닝담당자
    • 튜닝담당자 자동할당이 ‘No’일 경우 튜닝담당자를 직접 선택한다.
    이전 튜닝대상 선정 SQL 제외
    • ‘Yes’일 경우 해당 SQL점검팩에서 튜닝대상 선정이 된 적이 있는 SQL은 튜닝대상에서 제외한다.
    SQL점검팩
    • SQL점검팩
  3. 튜닝대상 선정 작업이 진행되고 완료되면 아래와 같은 메시지 박스가 팝업된다. 메시지 내용에 선정된 튜닝대상 건수를 확인할 수 있다.
  4. [Ok] 버튼을 클릭하면 그리드가 재조회 되며 튜닝상태, 튜닝번호 컬럼에서 확인이 가능하다.

튜닝대상 선정 개별 선택

검색된 SQL 중 개별 선택된 건에 대해 튜닝대상으로 선정한다. 개별 선택은 현재 페이지에서만 가능하다. [다음] 버튼을 클릭하여 페이지가 변경되면 선택된 체크박스가 해제된다.

  1. 튜닝대상 SQL에 대한 체크 박스를 클릭한다. 멀티 선택이 가능하다.
  2. 이후의 처리 과정은 ‘> 튜닝대상 선정 전체 선택’ 과 처리 과정이 동일하므로 해당 매뉴얼을 참조한다.

  1. [검색] 후 그리드에서 삭제할 SQL점검팩을 체크한 후 [삭제] 버튼을 클릭한다.
  2. 아래와 같은 메시지 박스가 팝업되고 [YES] 버튼 클릭 시 SQL점검팩이 삭제되었다는 메시지 박스가 팝업된다. SQL점검팩 삭제시 성능 영향도 분석 정보가 모두 삭제되므로 주의가 필요하다.
    튜닝대상으로 선정된 SQL이 존재하는 경우 아래와 같은 메시지 박스가 팝업되고, SQL점검팩을 삭제할 수 없다.

NOTE

튜닝대상이 선정되었을 경우 SQL점검팩을 삭제할 수 없다. 삭제 시 모든 데이터가 삭제되므로 실수로 인한 데이터 삭제를 방지하기 위한 목적이다.
