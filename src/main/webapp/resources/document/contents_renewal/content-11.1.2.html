11.1. 성능 최적화 관리

11.1.2. 성능 개선 관리

개요

성능개선요청 화면 또는 성능진단/분석 화면을 통해 요청된 SQL에 대하여 요청 -> 튜닝 -> 적용 등의 성능개선관리 프로세스를 처리한다.

개발자, DBA, 튜너, 관리자 모두 동일한 화면을 사용하지만 각 역할별 목록 대상은 아래와 같다.

그리드 클릭

  1. 그리드를 클릭한다.
  2. 그리드 클릭 시 성능 개선 관리 상세 (튜닝번호별) 페이지로 이동한다.

전체 선택

  1. ‘전체’ 체크 박스를 클릭하면 하위의 모든 체크 박스가 체크된다.
  2. 튜닝처리 이벤트 버튼을 클릭한다. (버튼 상세설명은 아래의 튜닝 처리 상태를 참고한다.)

개별 선택

  1. 튜닝대상 SQL에 대한 체크 박스를 클릭한다. 멀티 선택이 가능하다.
  2. 튜닝처리 이벤트 버튼을 클릭한다. (버튼 상세설명은 아래의 튜닝 처리 상태를 참고한다.)
튜닝 처리 상태

튜닝 담당자 지정

성능관리자(DBA) 권한을 가진 사용자만 튜닝 담당자 지정이 가능하다.

  1. 그리드를 선택한 후 [튜닝담당자 지정] 버튼을 클릭한다.
  2. 튜닝담당자 지정 팝업이 뜬다. 튜닝담당자를 선택 후 [담당자 지정] 버튼을 클릭한다.
  3. 튜닝담당자 지정이 완료 메시지창이 뜨며 담당자가 지정된다.

튜닝중 처리

성능관리자(DBA), 튜닝담당자 권한을 가진 사용자만 튜닝중 처리가 가능하다.
(성능 개선 관리 상세 > 개선 프로세스 [튜닝중 처리] 버튼도 동일하다.)

NOTE

진행상태가 튜닝대기 또는 적용반려 상태가 아니면 튜닝중 처리가 불가능하다.

  1. 그리드를 선택한 후 [튜닝중 처리] 버튼을 클릭한다.
  2. 그리드 컬럼의 진행상태가 튜닝대기에서 튜닝중으로 변경된다.

튜닝취소 처리

성능관리자(DBA), 튜닝담당자 권한을 가진 사용자만 튜닝취소 처리가 가능하다.
(성능 개선 관리 상세 > 개선 프로세스 [튜닝취소 처리] 버튼도 동일하다.)

NOTE

진행상태가 튜닝중이 아니면 튜닝취소가 불가능하다.

  1. 그리드를 선택 후 [튜닝취소 처리] 버튼을 클릭한다.
  2. 그리드 컬럼의 진행상태가 튜닝중에서 튜닝대기로 변경된다.

요청취소 처리

모든 사용자가 요청취소 처리를 할 수 있다.

NOTE

진행상태가 요청중이 아니거나 자신의 요청건이 아니면 튜닝취소가 불가능하다.

  1. [요청취소 처리] 버튼을 클릭한다.
  2. 그리드 컬럼의 진행상태가 요청에서 요청취소로 변경된다.
성능 개선 관리 상세

성능개선관리 튜닝목록을 선택하면 성능개선 상세화면으로 이동하며 성능개선과 관련된 모든 이벤트는 상세화면의 [개선 프로세스] 탭에서 처리한다.

[SQL요청 상세] 탭

요청내용을 보여주는 화면으로 개발자 요청, 성능분석/진단을 통한 요청 등 요청 유형(채널)에 따라 요청 상세 화면의 항목이 조금씩 다르게 보여 진다.
예를 들어 SQL진단 > PLAN변경을 통해 요청된 SQL은 PLAN 변경 전과 변경 후의 성능정보가 함께 요청 정보로 전달되어 SQL요청상세 화면에 보여짐으로 튜닝을 진행하는데 도움되는 정보로 활용할 수 있다.

SQL요청상세화면은 추가적으로 현재 요청에 대한 프로세스처리이력 조회, 튜닝이력 조회, SQL정보 조회 기능을 제공한다.

인덱스 변경

인덱스 변경 및 추가가 필요할 경우 등록한다.

  1. [인덱스 변경] 버튼을 클릭한다. 인덱스 요청 팝업창이 뜬다.
  2. OWNER, 테이블명을 입력 후 [검색] 버튼을 클릭한다.
  3. 체크박스 선택 후 [선택 반영] 버튼을 클릭한다. 하단그리드에 데이터가 반영된다.
  4. 하단 그리드 체크 박스 선택 후 [반영]을 클릭한다. 적용이 완료된다.
[개선 프로세스] 탭

성능개선관리에서 필요한 모든 이벤트를 처리하는 화면이다. 진행상태와 권한에 따라 화면이 다르다.

튜닝 사례 게시 등록

튜닝 사례 공유가 필요한 튜닝 건의 경우 사례 게시를 선택하고, 사례 제목을 입력 한 후 튜닝완료하면 자동으로 사례로 등록된다. SQL 튜닝 사례/가이드 메뉴에서 확인이 가능하다.(상단 우측 아이콘 클릭)

[SQL개선 이력] 탭

해당 튜닝 건에 대한 튜닝 이력을 보여주는 화면이다. 요청 -> 튜닝 -> 적용반려 -> 튜닝 -> 적용반려 -> 튜닝 -> 적용완료 와 같이 튜닝 프로세스가 순환되는 과정에서 튜너가 과거에 튜닝한 이력을 확인할 수 있다.
